Implementing Refresh Token logic requires a "Queue" system. When a 401 (Unauthorized) error occurs because a token has expired, you don't want to simply fail the request. Instead, you intercept it, call a refresh endpoint, update the token, and then retry the original failed request seamlessly.
Here is the updated api.client.js with the full implementation:
Updated Axios Client with Refresh Logic (src/services/api.client.refreshToken.tsx)


### How this works:
1.  **`_retry` Flag:** We attach this to the `originalRequest` so we don't end up in an infinite loop if the refresh token itself is expired.
2.  **`isRefreshing` Variable:** If five API calls fire at the exact moment the token expires, this ensures we only hit the `/refresh` endpoint **once**.
3.  **`failedQueue`:** While the new token is being fetched, the other four requests "wait" in this array. Once the token arrives, they are all resolved and re-sent automatically.
4.  **Security Fail-safe:** If the refresh call fails (meaning the user's session is totally dead), we clear `localStorage` and redirect them to the login page.

### Pro-Tip for React 19:
Since React 19 handles transitions and "Actions" more natively, you can combine this service with the `use` hook or `Suspense` for a very smooth UI experience where the user never even sees a loading spinner during the split-second token refresh.


## FIX Error for can not find 'process' in 'api.client.tsx'##

This error occurs because TypeScript doesn't recognize the global process object by default in a browser environment. While modern build tools (like Vite or Webpack) inject environment variables, TypeScript needs a "definition" to understand what process is.

Since you are building a React frontend, you have two main ways to solve this depending on your build tool.

Solution 1: The Standard Way (Recommended)
As the error message suggests, you need to provide the type definitions.

Install the types:

Bash
npm install --save-dev @types/node
Update your tsconfig.json:
Add "node" to the types array under compilerOptions. This tells TypeScript to include Node.js global definitions.

JSON
{
  "compilerOptions": {
    "types": ["node"],
    // ... other options
  }
}
Solution 2: The "Vite" Way (If you are using Vite)
If you created your React 19 project using Vite (which is the current standard), you actually shouldn't use process.env. Vite uses import.meta.env.

Change the variable access:
In your api.client.tsx, replace process.env.REACT_APP_API_URL with:

TypeScript
const baseURL = import.meta.env.VITE_API_URL || 'https://api.example.com';
Update your .env file:
Vite requires environment variables to be prefixed with VITE_ instead of REACT_APP_.

Plaintext
VITE_API_URL=https://your-api-url.com
Solution 3: The "Quick Fix" (Type Assertion)
If you want to bypass the error immediately without changing configurations, you can tell TypeScript that process exists globally by adding this at the very top of your api.client.tsx:

TypeScript
declare var process: {
  env: {
    NODE_ENV: string;
    [key: string]: string | undefined;
  };
};

const apiClient = axios.create({
  baseURL: process.env.REACT_APP_API_URL || 'https://api.example.com',
});
Which one should you choose?
If you are using Create React App (CRA): Use Solution 1.

If you are using Vite: Use Solution 2.

If you are in a Senior/Production environment: Use Solution 1 but ensure your tsconfig is strictly managed.
